# Market Intel — プロジェクトビジョン

## 一言で言うと

**対象（アプリ・人物・企業など）の情報と周辺環境を整理し、今の立ち位置・過去の推移・次の一手を導くインテリジェンスツール。**

---

## 何を解決するのか — 3つのコア機能

### 1. 現在地の把握 — 「今どこにいるのか」

対象と周辺環境を一画面で整理する。

- **マクロ環境**: 対象が属する市場や業界全体のトレンド、ニュース、季節性
- **競合の動向**: 比較対象の動き（順位、評判、広告、業績）
- **ユーザー/市場の反応**: レビュー、SNSセンチメント、オッズ、出来高など末端の反応

> 「今、追い風の中にいるのか、逆風の中にいるのか」が一目でわかる。

### 2. 推移の確認 — 「過去からどう変わってきたか」

時系列でデータを追い、変動の因果関係を紐付けて振り返れるようにする。

- いつ、何が起きて、何が変わったか
- そのアクションが本当に効いたのか、外部要因だったのか
- 因果ログとして蓄積し、後から振り返れる

> 「あの施策が効いたのか、市場の追い風だったのか」を区別できる。

### 3. 次の一手の予測 — 「何をすればより良くなるか」

蓄積された因果パターンから、成功確率の高いアクションを提示する。

- 過去に似た状況で何が効いたか
- 信頼度の高いパターン（繰り返し検証されたもの）を優先表示
- やっても効果がなかったパターンを自動淘汰

> 「勘」ではなく「蓄積された根拠」に基づく意思決定を支援する。

---

## なぜこの3つなのか

```
現在地がわからない → 判断の前提がない
推移がわからない   → 何が効いたか学べない
予測がない         → 毎回ゼロから考える

この3つが揃って初めて「インテリジェンス」になる。
データ収集だけでは情報、分析だけでは知識。
行動に繋がる示唆まで届けて初めて知恵（Intelligence）。
```

## 4層モデル — なぜ汎用に効くのか

このツールの方法論の核は **4層収集モデル** にある。
対象が何であっても、周辺環境は同じ4層で構造化できる。

| 4層モデル | 意味 | アプリ | 株式 | 競馬 |
|-----------|------|--------|------|------|
| **L1 マクロ** | 対象を取り巻く大きな潮流 | 市場トレンド、業界ニュース | 金利・為替・セクター指数 | JRA売上、馬場状態、天候 |
| **L2 競合** | 比較対象の動き | 競合アプリの順位・レビュー・広告 | 同業他社の決算・株価 | 各馬の成績・調教・騎手 |
| **L3 ユーザー** | 末端の反応（先行指標） | アプリレビュー、SNSセンチメント | SNS・掲示板の話題量 | オッズ推移、SNS予想の偏り |
| **L4 因果** | 蓄積された「こうしたらこうなった」 | イベント→トレンド異常値の相関 | 決算発表→翌日値動き | 馬場×血統×距離→回収率 |

**対象を差し替えるだけで、同じ収集・分析・学習の構造が動く。**

## ユースケースの広がり

4層モデルが成立する対象なら何にでも適用できる。
ただし最初から手を広げず、メインで完成度を高めてからサブに展開する。

実データで因果学習を回し、ツールの価値を検証する。

## ダッシュボードの構成方針

### 基本構造: 3タブ × カスタマイズ可能なウィジェット

タブ構成（現在地/推移/次の一手）は全ユースケース共通。
各タブ内に **何を表示するかはドメイン設定とユーザーの選択で変わる**。

```
┌─────────────────────────────────────────────────┐
│  Market Intel              [対象: モンスト ▼]    │
├──────────┬──────────┬──────────┬────────────────┤
│  現在地   │  推移     │  次の一手 │               │
│ Position │ History  │ Actions  │               │
└──────────┴──────────┴──────────┴────────────────┘

[現在地] タブ
  - スコアカード: 対象の主要KPI
  - ポジション: 競合・比較対象との相対位置
  - 環境サマリー: マクロトレンドの追い風/逆風インジケータ

[推移] タブ
  - タイムライン: 主要指標の時系列チャート
  - イベントオーバーレイ: アクション・競合の動き・市場イベントを重畳表示
  - 因果ハイライト: 「この変動はこれが原因の可能性が高い」を注釈表示

[次の一手] タブ
  - 推奨アクション: 信頼度の高い因果パターンに基づく提案
  - リスク警告: 検出された逆風パターン
  - パターン一覧: 蓄積された因果ログ（フィルタ・信頼度順）
```

### カスタマイズの考え方

ユースケースによって「見たいもの」は違う。
タブの骨格（現在地/推移/次の一手）は固定しつつ、中身をカスタマイズ可能にする。

```
ドメイン設定 (config/domains/*.json)
  → どのコレクターを使うか（データソースの選択）
  → どの指標を主要KPIとするか
  → 季節パターン辞書

ユーザー設定 (ダッシュボード上)
  → 表示するウィジェットの選択・並べ替え
  → 注目指標のピン留め
  → 因果メモの表示フィルタ（信頼度、パターンタイプ）
```

ドメイン設定がデータの形を決め、ユーザー設定が見え方を決める。
この分離により、同じドメインでも人によって異なるダッシュボードになる。

## 差別化 — 既存ツールとAI検索に対して

### 既存ツールとの関係

各層に強い専業ツールは存在する（Sensor Tower, Brandwatch, TradingView等）。
しかし **L1〜L4を統合したツールは存在しない**。

| | L1 マクロ | L2 競合 | L3 ユーザー | L4 因果学習 |
|---|:---:|:---:|:---:|:---:|
| Sensor Tower / AppTweak | - | ◎ | ○ | - |
| Crayon / Klue | - | ◎ | - | - |
| Brandwatch / Brand24 | △ | - | ◎ | - |
| Exploding Topics | ○ | - | - | - |
| TradingView / Koyfin | ○ | ○ | - | - |
| **Market Intel** | ○ | ○ | ○ | **○** |

各層の個別データ品質では専業に勝てない。
**勝負するのは「統合」と「因果学習（L4）」** — ここが空いている。

将来的には Sensor Tower や MMPP 等の外部APIと連携し、
L1〜L3のデータ品質を底上げする拡張を想定。
その場合もL4の因果学習エンジンがこのツール固有の価値になる。

### AI検索（ChatGPT / Perplexity等）との関係

AI検索は **ステートレス**。その場の知識で答え、毎回リセットされる。
このツールは **ステートフル**。データと因果を蓄積し続け、使うほど賢くなる。

| | AI検索 | Market Intel |
|---|---|---|
| 「市場の競合状況は？」 | ◎ その場で良い回答 | 同等かそれ以下 |
| 「先月と比べてどう変わった？」 | × 過去の状態を知らない | ○ 時系列で追っている |
| 「あの施策は効いたのか？」 | × あなたの施策を知らない | ○ 因果ログに蓄積されている |
| 「今の状況で何をすべき？」 | △ 一般論は出る | ○ 過去の類似パターンから根拠付き |
| 1ヶ月後に同じ質問 | 毎回ゼロから | 前回の続きから |

AI検索は競合ではなく **組み込む側**。

```
Market Intel の蓄積データ（時系列 + 因果ログ）
    ↓
LLM に渡して解釈・提案を生成
    ↓
蓄積データというコンテキストがあって初めて
LLM が具体的な示唆を出せる
```

**このツールの本質的な価値は「蓄積」にある。**
データソースは後から良いものに差し替えられるし、解釈エンジンも進化する。
でも蓄積された因果ログは時間をかけないと作れない。

## 因果学習の設計方針 — 自動メモの限界と対策

### 現状の自動メモでできること

5つのパターンを自動検出し、信頼度を付与して蓄積：

- **異常値×イベント**: Z-score異常 + イベント時間近接 → 相関の検出
- **トレンド変動**: 移動平均の前期比変動 → 変化の検出
- **レビュー急変**: スコア/件数の月次変動 → 変化の検出
- **ニュース相関**: ニュース + 異常値の時間近接 → 相関の検出
- **季節パターン**: 月ベースのカレンダーマッチ

検出 → 自動検証 → 信頼度更新 のループは手動操作なしで回る。

### 自動メモだけでは足りない点

致命的に欠けているのは **「自分が何をしたか」の情報**。

```
自動で検出できる:
  「RPGトレンドが先週12%上昇した」
  「モンストのレビューが4.2→3.8に急落した」

自動では検出できない:
  「火曜にバグ入りアプデを出した」    ← これが原因
  「先月からガチャの排出率を変えた」  ← これが原因
  「競合が有名VTuberとコラボした」   ← RSSに出ないこともある
```

自分のアクションを知らないのに「こうしたらこうなった」は言えない。
自動メモは「こうなった×こうなった」（相関）であって因果ではない。

### 解決策: 検出は自動、文脈は最小限の手動入力

完全自動か完全手動かではなく、**ハイブリッド**。

```
1. 自動検出（現行のまま）
   「モンストのレビューが急落した」

2. システムが文脈を聞く（新機能）
   「この前後で何かしましたか？」
   [アプデ出した] [イベント開始] [広告変更] [特になし]
          ↓ 選択後
   簡単なメモを追記できる: 「v2.3.1 課金周りのバグ」
          ↓ 選択肢にない場合
   自由入力で記帳: 「競合がTVCM開始した」

3. 因果メモに昇格
   「バグ入りアプデ(v2.3.1) → レビュー急落」(信頼度 0.70)

4. 次回類似パターンで自動検証
   同様のアプデ後にレビューが下がれば信頼度↑、下がらなければ↓
```

入力の3段階:
- **ワンタップ**: 選択肢を選ぶだけ（最も軽い、これだけでも因果成立）
- **選択肢＋メモ**: 選んだ上で一言補足（精度が上がる）
- **自由記帳**: 選択肢にない出来事を自分で書く（想定外の因果を捕捉）

どの段階でも記録は残る。重要なのは **入力しなくても自動メモは回り続ける** こと。
文脈入力は「あればより良い」であって「ないと動かない」ではない。

## 既知の課題と次のアクション

### 最優先: 因果ログの永続化

**「蓄積が価値」と言っているのに、蓄積が永続化されていない。**

| データ | 現在の保存先 | リロードで |
|---|---|---|
| パターンの重み(学習率) | localStorage | 残る |
| 学習設定(閾値等) | localStorage | 残る |
| **手動メモ** | **React state のみ** | **消える** |
| **自動検出メモ** | **React state のみ** | **消える** |

`CausationView.jsx` で `useState` に入れているだけで永続化していない。
ツールの中核価値が揮発性メモリにしかない状態。

localStorage 自体もブラウザのデータ削除で消える。
Phase 4 で SQLite 移行を計画しているが、それ以前にまず保存すらされていない。

**対応**: まず localStorage に保存 → 次に IndexedDB or SQLite に移行。

### 高: 自動検証の循環論法を修正

`autoMemo.js` のバリデーターのうち2つが事実上の自己承認:

```js
// validateAnomalyEvent — 信頼度0.6以上なら無条件で「的中」
return memo.confidence >= 0.6 ? 'confirmed' : null

// validateNewsCorrelation — 信頼度0.55以上なら無条件で「的中」
return memo.confidence >= 0.55 ? 'confirmed' : null
```

データから検出したものを同じデータで「的中」と判定 = 学習ではなく自己強化。
信頼度が上がり続けるだけで、実際に正しいかの検証になっていない。

**対応**: 検出時点と検証時点のデータを分離する。
「異常値×イベントの相関を検出した後、次の類似イベント時に同じパターンが再現するか」
を見て初めて confirmed にする設計に変更。

### 高: 文脈入力UIの実装

前述のハイブリッド方式（ワンタップ選択肢 + メモ + 自由記帳）が未実装。
因果学習の差別化ポイントなのに、UIもデータモデルも存在しない。

**対応**: 自動検出メモに対して文脈を紐付けるUIコンポーネントを新規作成。
データモデルに `userContext` フィールドを追加。

### 中: ダッシュボードのドメイン非依存化

フレームワーク層（domain.js, layers.js, collector-registry.js）はドメイン非依存だが、
ダッシュボードは約80%がゲーム市場専用:

- タブ名: 「アプリ情報」「ユーザー」「業界・イベント」
- コンポーネント: AppInfoView, CorporateView（ゲーム企業構造）
- 定数: 「ガチャ」「コラボ」「シーズン」がハードコード
- 検索: モンスト、パズドラ等のプリセット

ビジョンでは「現在地/推移/次の一手」の3タブ構成を掲げているが、
現在の実装とは構造が根本的に異なる。

**対応**: ドメイン設定からタブのラベル・表示ウィジェット・KPI定義を
読み込む仕組みに段階的にリファクタリング。

### 中: インフルエンサードメインの設計

メインに昇格したが、config/domains/ にも collectors/ にも何もない。

**対応**: YouTube Data API / Social Blade 等のデータソース選定、
`config/domains/influencer.json` のスキーマ設計から着手。

## 次セッションの着手順

```
1. 因果ログの永続化（localStorage → 最低限の保存）
   ↓ これがないと蓄積が始まらない
2. 自動検証ロジックの修正（循環論法の排除）
   ↓ 学習が正しく回るようにする
3. 文脈入力UI（ワンタップ + メモ + 自由記帳）
   ↓ 相関→因果への昇格手段を作る
4. ダッシュボードの3タブ再構成（現在地/推移/次の一手）
   ↓ ビジョンと実装を一致させる
5. インフルエンサードメイン設計
```

1〜3 は因果学習エンジンの基盤。ここが動かないとツールの価値が成立しない。
4〜5 はその上に載せるUI/ドメインの話。

---

## 開発方針: 完成度優先、焦らない

機能を広げるより、因果学習の基盤を固めることを優先する。

- 4層統合 + 因果学習は現時点で空白地帯。すぐ追いつかれる領域ではない
- 仮に類似ツールが出ても、因果ログの蓄積は時間がかかる。先に回し始めた側が有利
- 中途半端な完成度で広げると「既存ツールの劣化版」で終わる
- 自分たちが毎週使って因果が溜まる状態を先に作ることが最優先

## 将来構想: 行動分析シミュレーションとの連携

別プロジェクトで進行中の行動分析シミュレーションと連結し、双方の精度を高める。

### データの流れ

```
Market Intel                        行動分析シミュレーション
┌──────────────┐                   ┌──────────────────────┐
│ 4層の環境データ │ ── 初期条件 ──→ │ シミュレーション実行   │
│ 因果ログの蓄積  │ ── パラメータ →│ 行動パターンの予測     │
│              │                   │                      │
│ 「次の一手」の │ ←─ 予測結果 ──  │ 「この行動をとると    │
│  精度向上     │                   │  こうなる」の推定     │
│              │                   │                      │
│ 実績データの   │ ── フィードバック→│ モデルの精度改善      │
│ 蓄積・照合    │                   │                      │
└──────────────┘                   └──────────────────────┘
```

### 相互に高め合う関係

**Market Intel → シミュレーション**:
- 因果ログ（実績ベースの「こうしたらこうなった」）をモデルのパラメータに活用
- 4層の環境データ（マクロ・競合・ユーザー）をシミュレーションの初期条件に
- 蓄積が増えるほどシミュレーションの入力が豊かになる

**シミュレーション → Market Intel**:
- 「この行動をとったらどうなるか」のシナリオ検証結果を「次の一手」タブに反映
- 複数シナリオの比較で行動提案の信頼度を補強
- シミュレーション結果を Market Intel が実績と照合し、予測モデルを改善

### 連携インターフェースの方針

今の段階では構想のみ。因果学習エンジンが固まってから設計する。

想定する接続ポイント:
- **因果ログの公開API**: 蓄積されたパターン（信頼度付き）をJSON/APIで提供
- **環境データの公開API**: 4層の最新データをスナップショットで提供
- **予測結果の受け入れ**: シミュレーション結果を因果メモとして取り込み、実績と照合

## 成功の定義

このツールが「うまくいった」と言えるのは：

1. **使い続けられる** — 定期的にダッシュボードを開いて意思決定に使う
2. **蓄積が効く** — 時間とともに因果ログの精度が上がっている実感がある
3. **説明できる** — 「対象の立ち位置と次にやるべきことを教えてくれるツール」と一文で言える
4. **対象を変えても動く** — ドメイン設定を差し替えれば、別の対象でも同じ価値を得られる
