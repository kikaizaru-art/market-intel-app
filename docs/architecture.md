# Multi-Domain Intelligence Framework — Architecture

## 設計思想

本フレームワークの本質は **「環境変数を多層的に収集し、因果関係を蓄積・学習する」** ことにある。
対象ドメインを差し替えても同じ構造が動く — これがマルチドメイン対応の核心。

```
┌─────────────────────────────────────────────────────┐
│              Domain Config Layer                     │
│  game-market.json │ stock.json │ keiba.json          │
└────────┬──────────┴─────┬──────┴─────┬──────────────┘
         │                │            │
         ▼                ▼            ▼
┌─────────────────────────────────────────────────────┐
│            4-Layer Collection Engine                 │
│  L1 マクロ │ L2 競合 │ L3 ユーザー │ L4 因果関係     │
└────────┬──────────┴─────┬──────┴─────┬──────────────┘
         │                │            │
         ▼                ▼            ▼
┌─────────────────────────────────────────────────────┐
│         Domain-Agnostic Analysis Engine              │
│  異常値検出 │ トレンド分析 │ 因果候補抽出             │
└────────┬──────────┴─────┬──────┴────────────────────┘
         │                │
         ▼                ▼
┌─────────────────────────────────────────────────────┐
│         Causal Log Auto-Accumulation                 │
│  自動メモ生成 → 自動検証 → 信頼度学習 → パターンDB   │
└─────────────────────────────────────────────────────┘
```

## 1. 4層モデル — ドメイン間マッピング

フレームワークの骨格は **4つの環境変数レイヤー** で構成される。
各ドメインでは収集対象が変わるだけで、分析ロジックは共通。

| レイヤー | ゲーム市場 (現行) | 株式 | 競馬 |
|----------|-------------------|------|------|
| **L1 マクロ** | ジャンル別Google Trends、業界ニュース、CPI | 金利・為替・セクター指数、経済指標 | JRA全体売上、馬場状態、天候 |
| **L2 競合** | 他社アプリのストア順位・レビュー・広告出稿 | 同業他社の決算・IR・出来高 | 各馬の過去成績・調教データ・騎手成績 |
| **L3 ユーザー** | アプリレビュー、SNSセンチメント | SNSセンチメント・掲示板の話題量 | オッズ推移・SNS予想の偏り |
| **L4 因果関係** | イベント×トレンド異常値の相関 | 決算発表→翌日値動き、規制ニュース→セクター反応 | 雨馬場×血統×距離→回収率パターン |

### 各レイヤーの役割

- **L1 マクロ**: 市場全体の潮流。個別の意思決定の前提条件を形成
- **L2 競合**: 直接比較可能な対象の動向。差別化と機会の発見
- **L3 ユーザー**: 末端の反応と感情。先行指標として最も早く変化を捕捉
- **L4 因果関係**: 上記3層から自動抽出される因果パターンの蓄積。時間とともに精度向上

## 2. ドメイン設定スキーマ

各ドメインは `config/domains/{domain}.json` で定義される。
同一スキーマに従うことで、フレームワークがドメインを問わず動作する。

```jsonc
{
  // ドメイン識別
  "domain": "game-market",
  "name": "ゲーム市場分析",
  "version": "1.0.0",

  // 4層の定義 — 各レイヤーにコレクターとデータソースを紐付け
  "layers": {
    "macro": {
      "label": "マクロ環境",
      "collectors": ["google-trends", "news-rss"],
      "sources": { ... }
    },
    "competitor": {
      "label": "競合動向",
      "collectors": ["store-reviews", "meta-ads"],
      "sources": { ... }
    },
    "user": {
      "label": "ユーザー反応",
      "collectors": ["store-reviews"],
      "sources": { ... }
    },
    "causal": {
      "label": "因果関係",
      "autoMemo": true,
      "patterns": [ ... ]
    }
  },

  // 分析対象（ゲームならtitles、株ならtickers、競馬ならhorses）
  "targets": [ ... ],

  // 季節パターン辞書（ドメイン固有）
  "seasonalPatterns": [ ... ],

  // データ保存先プレフィックス
  "dataPrefix": "game"
}
```

## 3. コレクターレジストリ

コレクターはドメインに依存しない **プラグイン** として登録される。
ドメイン設定が「どのコレクターを使うか」を宣言し、
レジストリが対応するモジュールを解決・実行する。

```
CollectorRegistry
  ├── register(name, collectorFn)    # コレクター登録
  ├── resolve(domainConfig)          # ドメイン設定から必要なコレクターを解決
  └── runAll(domainConfig)           # 全コレクターを並列実行（fail-safe）

コレクターの契約:
  Input:  { sources, targets, options }  ← ドメイン設定から注入
  Output: { layer, data, metadata }      ← 統一出力フォーマット
```

### 既存コレクターのマッピング

| コレクター名 | 現行モジュール | ゲーム | 株式 | 競馬 |
|-------------|---------------|--------|------|------|
| `google-trends` | `collectors/trends.js` | キーワードトレンド | 銘柄名トレンド | レース名トレンド |
| `store-reviews` | `collectors/store.js` | アプリレビュー | — | — |
| `news-rss` | `collectors/news.js` | ゲーム系RSS | 経済ニュースRSS | 競馬ニュースRSS |
| `meta-ads` | `collectors/meta-ads.js` | 広告出稿 | — | — |
| `stock-price` | (新規) | — | 株価API | — |
| `financial-data` | (新規) | — | 決算・IR | — |
| `jra-results` | (新規) | — | — | レース結果 |
| `odds-tracker` | (新規) | — | — | オッズ推移 |
| `sns-sentiment` | (新規) | SNS分析 | SNS分析 | SNS分析 |

## 4. ドメイン非依存の分析パイプライン

```
Raw Data (per collector)
    │
    ▼
┌─────────────────────┐
│   Normalizer        │  ドメイン固有の生データ → 統一時系列フォーマット
│   (per collector)   │  { date, metric, value, source, layer }
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Anomaly Detector  │  Z-scoreベース (既存: analyzers/anomaly.js)
│   (domain-agnostic) │  閾値はドメイン設定で調整可能
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Trend Analyzer    │  移動平均 + 前期比 (既存: analyzers/trend.js)
│   (domain-agnostic) │  window/lagはドメイン設定で調整可能
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Causation Finder  │  異常値×イベントの時間近接度 (既存: analyzers/causation.js)
│   (domain-agnostic) │  windowDaysはドメイン設定で調整可能
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Auto-Memo Engine  │  パターン検出 → 自動メモ → 検証 → 学習
│   (domain-agnostic) │  パターンタイプはドメイン設定で拡張可能
└─────────────────────┘
```

### 既存アナライザーの抽象度

現行の `src/analyzers/` は既にドメイン非依存に近い設計になっている:

- **`anomaly.js`**: 数値配列を受け取りZ-scoreで異常値を返す。入力がジャンルでも銘柄でも馬でも動作する
- **`trend.js`**: 移動平均と前期比。時系列データなら何でも適用可能
- **`causation.js`**: 日付ベースの近接度計算。イベントの意味はドメインに委譲

唯一ドメイン依存が強いのは **`autoMemo.js`** の以下の部分:
- 季節パターン辞書 → ドメイン設定に外出し
- パターンラベル → ドメイン設定で上書き可能に
- 検証ロジック → バリデーターをドメインごとに登録可能に

## 5. 因果ログ自動蓄積エンジン — フレームワークの中核

**全ドメインで最も価値を生むのがこの層。**

```
日々の自動実行:
  1. コレクターが全レイヤーのデータを収集
  2. アナライザーが異常値・トレンド変動を検出
  3. Auto-Memo エンジンがパターンを自動生成
  4. 検証エンジンが過去メモの予測を実績と照合
  5. 学習エンジンが信頼度重みを更新
  6. パターンDBに蓄積

蓄積が進むほど:
  - 信頼度の高いパターンが浮上
  - 偽陽性パターンが自動淘汰
  - ドメイン固有の「勝ちパターン」が可視化
```

### ドメイン別の因果ログ例

**ゲーム市場**:
```
[確信度 0.87] ガチャイベント開始 → RPGトレンド急上昇 (±3日以内、過去12回中10回的中)
[確信度 0.72] 競合の大型アプデ → 自社レビュースコア低下 (間接効果、反応遅延5-7日)
```

**株式**:
```
[確信度 0.83] 決算発表(上方修正) → 翌営業日+3%以上 (過去8回中7回的中)
[確信度 0.65] 米国CPI発表 → セクター全体の出来高急増 (±1日以内)
[確信度 0.41] SNSでの炎上 → 株価下落 (ノイズ多め、要条件絞り込み)
```

**競馬**:
```
[確信度 0.91] 重馬場 × ステイゴールド産駒 × 2000m以上 → 回収率 142% (n=23)
[確信度 0.78] 前走4角3番手以内 × 今回距離短縮 → 連対率 38% (n=45)
[確信度 0.55] SNS予想偏り(単勝支持率>40%) → 実際の着順と逆相関
```

## 7. 統一データモデル

全ドメインのデータを共通モデルに正規化することで、
アナライザーとダッシュボードがドメインを意識せずに動作する。

### 時系列データ (L1-L3 共通)

```javascript
{
  date: "2025-01-15",      // ISO日付
  metric: "RPG",           // ゲーム: ジャンル名 / 株: 銘柄コード / 競馬: レース条件
  value: 78.5,             // 正規化された数値 (0-100 or 生値)
  source: "google-trends", // コレクター名
  layer: "macro",          // L1-L3
  metadata: {}             // ドメイン固有の追加情報
}
```

### 因果メモ (L4)

```javascript
{
  id: "auto_xxx",
  date: "2025-01-15",
  event: "...",                        // 検出されたパターンの説明
  target: "モンスト",                   // ゲーム: アプリ名 / 株: 銘柄 / 競馬: 馬名
  layer: "macro",                      // どのレイヤーで検出されたか
  impact: "positive|negative|neutral",
  memo: "...",                         // 詳細説明
  auto: true,
  patternType: "anomaly_event",
  confidence: 0.82,
  signals: [],
  domain: "game-market",              // どのドメインで生成されたか
  _validation: {}
}
```

## 8. ドメイン拡張の手順

新しいドメインを追加する際の手順:

### Step 1: ドメイン設定ファイルを作成
```bash
config/domains/new-domain.json
```

### Step 2: 必要なコレクターを実装
```javascript
// src/collectors/domains/new-domain/price-fetcher.js
export default {
  name: 'price-fetcher',
  layer: 'macro',
  async collect({ sources, targets }) {
    // データ収集ロジック
    return { layer: 'macro', data: [...], metadata: { ... } }
  }
}
```

### Step 3: コレクターをレジストリに登録
```javascript
// src/framework/collector-registry.js
registry.register('price-fetcher', priceFetcher)
```

### Step 4: (任意) ドメイン固有のバリデーターを追加
```javascript
// src/framework/validators/new-domain.js
export function validateCustomPattern(memo, data) {
  // ドメイン固有の検証ロジック
  return 'confirmed' | 'rejected' | null
}
```

### Step 5: ドメインを有効化
```bash
# config/domains/active.json の domain を変更
# または環境変数 DOMAIN=new-domain npm run collect
```

## 9. 冷静に見ておくべき設計判断

### やること
- **分析基盤の共通化**: 異常値検出・トレンド分析・因果候補抽出はドメイン非依存
- **因果ログの長期蓄積**: 人間がやると続かない記録をAIが自動で回し続ける
- **根拠の可視化**: 予測精度よりも「なぜそう判断したか」が蓄積されること自体が価値
- **負けパターンの排除**: 信頼度が下がり続けるパターンを自動淘汰

### やらないこと
- **予測の確実性を謳うこと**: 株も競馬も「勝率を上げる」ことはできるが確実ではない
- **ドメイン固有ロジックの過度な抽象化**: 共通化できない部分は素直にドメイン固有に書く
- **全ドメイン同時開発**: まずゲーム市場で完成度を高め、次にデータが整備されている競馬、最後に株式
